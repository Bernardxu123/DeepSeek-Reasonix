三、核心架构 双模式运行
                    ┌──────────────┐
                    │  tauri-bridge │
                    └──────┬───────┘
                           │
           ┌───────────────┴──────────────┐
           │                              │
    mode=MODEL                          mode=server
    (Vite 开发)                          (CLI 生产)
           │                              │
    ┌──────┴──────┐              ┌───────┴───────┐
    │ Mock 模拟    │              │  SSE / REST   │
    │ 打字机/工具  │              │  真实 CLI 后端 │
    └─────────────┘              └───────────────┘
Mock 模式: tauri-bridge.ts 检测 reasonix-mode meta 为占位符 __REASONIX_MODE__ → 使用内置 mock 数据
Server 模式: meta 被 CLI 替换为真实值 → 连接 SSE /api/events?token=xxx + REST API
SSE → IncomingEvent 事件转换层
CLI 后端发送的 DashboardEvent (assistant_delta, tool_start 等) 被 sseToIncoming() 转换为前端 reducer 可消费的 IncomingEvent (model.delta, tool.preparing 等)。

RPC → REST 命令映射

五、已知问题 & 后续 TODO 1. SSE 连接不稳定 (P0)
/api/events 连接建立后出现 ERR_CONNECTION_RESET，随后服务器进程挂掉。可能原因：

handleEvents 内部订阅回调抛异常
ctx.subscribeEvents 在 standalone 模式下未就绪
定位方向: 检查 src/server/api/events.ts:writeEvent 异常处理，或在 src/server/index.ts SSE 路由处加 try/catch。

2. 主题自动加载 (P1)
首次加载时 localStorage 为空，main.tsx 已处理默认 dark 主题。但如果用户切换主题后 $settings 事件再次发射，主题态可能被覆盖。确认 App() 中的 useEffect([theme, themeStyle]) 是否正确同步。

3. new_chat 后会话列表不刷新 (P1)
POST /api/sessions/new 成功后前端未重新获取会话列表。需在 serverRpc 的 new_chat 分支加 emitEvent({ type: "$sessions", ... }) 重新获取。

4. 手机端真机调试 (P2)

七、接替者需要知道的
tauri-bridge.ts 是整个桥接层的核心 — 所有 @tauri-apps/* import 都指向它。修改 App.tsx 前先理解 bridge 的双模式逻辑。

styles.css 的移动端部分在文件末尾 — 搜索 MOBILE RESPONSIVE 定位。PC 和移动端在一起维护，不要拆散。

不要修改 dist/ — 它是构建输出。每次改完 src/ 后运行 npm run build 重新生成。

服务器 assets.ts 的 loadCss() 已改为优先加载 dist/app.css — 确保 dashboard 先构建好再启动 CLI。

__REASONIX_TOKEN__ / __REASONIX_MODE__ 是生产服务器替换的占位符，Vite 开发模式通过 transformIndexHtml 绕过它们。

SSE 事件格式差异 — CLI 后端发送 DashboardEvent (kind-based)，bridge 的 sseToIncoming() 转换到前端 IncomingEvent (type-based)。两边不对齐就在这里补。

项目有 REMEMBERED 反馈 — web-update-cli-not-desktop：Web 版本更新由 CLI 管理，不需要桌面 updater。

八、2026-05-20 后续计划修订：桌面端 Web 优先

当前最新策略：先把桌面端 Web 在真实 CLI Server 模式下跑通并稳定下来，再做手机端优化。标准流程是：

```powershell
cd D:\AI\workspace\dashboard
npm run build
cd D:\AI\workspace
npx tsx src/cli/index.ts code --open-dashboard
```

启动后会进入类似 `http://127.0.0.1:62197/?token=xxxx` 的地址。端口是动态分配的，`token` 是本次 dashboard 的访问令牌。后续调试、验收、截图和问题定位都要优先使用这个 URL；`npm run dev` 的 `http://127.0.0.1:3000` 只作为 Mock 视觉预览，不代表真实 CLI 联调结果。

阶段 A：桌面端 Web 跑通
- token URL 打开后 3 秒内进入完整 UI，不停留在 `loading…`。
- `tauri-bridge` 必须识别为 `mode=server`，并拿到 `reasonix-token`。
- 初始 `$tab_opened`、`$settings`、`$sessions`、`$ready` 事件全部进入当前 tab。
- 输入短消息后，用户消息、`assistant_delta`、`assistant_final`、`busy-change` 能完整流转。
- 点击停止能调用 `POST /api/abort`，并让 UI 回到可输入状态。
- 覆盖一次工具卡片和一次确认/选择弹窗，确认用户操作能回传到 `POST /api/modal`。

阶段 B：桌面端协议与稳定性补齐
- `session_load` 改为 `POST /api/sessions/:name/switch`。
- `session_delete` 改为 `DELETE /api/sessions/:name`。
- `new_chat` 成功后不要依赖返回 `name`；应重新 `GET /api/sessions` 并刷新会话列表。
- `/api/events?token=...` SSE 建立后后端不能崩溃，断线重连不能重复创建 turn 或重复追加消息。
- 所有 POST/DELETE 都必须通过 `x-reasonix-token` header 发送 token，不能只依赖 URL query。

阶段 C：桌面端异常处理
- token 无效：显示“链接已过期，请重新从 CLI 打开”。
- SSE 断开：显示“正在重连”，重连成功后自动恢复。
- 后端退出：显示“CLI 已停止”，禁用发送。
- `/api/submit` 失败：保留用户输入，不吞掉草稿。
- 构建产物缺失：CLI 启动时给明确提示，避免白屏。

阶段 D：桌面端验收矩阵与自动化
- 首屏加载：token URL 3 秒内进入完整 UI。
- 会话列表：能加载、切换、新建、删除非当前会话。
- 消息发送：连续 3 轮消息不丢、不重、不乱序。
- REST：submit / abort / sessions / modal 全部返回预期。
- 安全：无 token 访问 HTML/API 被拒绝，POST 必须 header token。
- 构建：`dashboard/dist/app.js` 与 `dashboard/dist/app.css` 存在并被 CLI 加载。
